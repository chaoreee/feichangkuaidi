## 0. 数据与口径

- 语料：67 局 `reports/*.compact.log`（N=67，Iter 32.5 真实基线，clientVersion=iter31）。
- 字段语义（`client/main.py:326,333`）：`node=`（me 节点）→ compact `n=`；`oppNode=`（对手节点）→ compact `on=`。**这是本次勘误的核心。**
- 路线核算（对齐 Iter 34 §1.2，纯移动损耗口径）：
  - 帧 `frames = ceil(ceil(distance × ROUTE_TIME_COEF[rt]) / 1000)`（`rules.frames_on_edge`，base_move=1000、无天气）
  - 损耗 `loss = frames × FRESHNESS_LOSS_MOVE[rt]`
  - 仅计移动、不含处理/验核停靠/天气/急策；实际与最优同口径故 Δ 公平（共享处理站 S13/S14 相消）。
- 最优路 = Dijkstra 按"帧数"边权 start→terminal（Iter 34 证三准则重合）。
- 工具：新模块 `analysis/route_audit.py`（复用 `core.rules` + `core.pathfind` + `analysis.compact.parse_compact` + `analysis.opponent_classifier`），11 项单测；`reports/route_audit.json` / `route_audit.md`。

## 1. 全量路线重建

### 1.1 me 路线是固定点

67 局中 **64 局** me 走完全相同的山路绕行：

```
S01 → S06 → S08 → S10 → S13 → S14 → S15
```

其余 3 局是未交付局（2 局卡在 S14、1 局卡在 S08，即 Iter 33 风暴 / 设卡卡死类）。**me 的路线与对手类无关、跨局近乎常量**——与 CLAUDE.md「我方 play 是固定点」一致。

### 1.2 勘误的勘误：`on=` ≠ me 途经节点

Iter 34 勘误（`docs/iter34_route_lever_analysis.md` §勘误）从 compact.log 重建 me 路线为 `S01→S02→S03→S06→S08→...`，据算 +64 帧 / +7.43 损耗。**该重建错误**：S02、S03 来自 compact 的 `on=` 字段（对手节点），非 me 途经。代表性败局 `match_..._2696_vs_2986` 的 compact 逐行：

```
F r1  n=S01 ... on=S01     # me=S01, opp=S01
A r7  MOVE target=S06      # me 发往 S06
F r43 on=S02               # 对手到 S02（非 me）
F r82 on=S03               # 对手到 S03（非 me）
F r86 n=S06                # me 到 S06（直连山路边 S01-S06:44:M，79 帧）
```

me 从 S01 直发 S06（山路边 `S01-S06:44:M`，79 帧，r7→r86），**不经 S02/S03**。S02/S03 是对手的 `on=` 轨迹。

## 2. 绕行动机归因

me 的 off-optimal 节点固定为 **S06、S08**（S10/S13/S14/S15 在最优路上）：

- **S06（MOUNTAIN）**：领 ICE_BOX（66/67 局）。冰鉴 +10 端鲜度、救好果。
- **S08（MOUNTAIN）**：领 SHORT_HORSE（64/67 局）+ 顺路 CLAIM_TASK（S08 是任务节点）。

绕行**不是被障碍逼路**：r1 的 `CLEAR target=S06` / `SQUAD_CLEAR target=S08` 是 me 主动清这两个山地节点的处理障碍（S06/S08 本身有 PROCESS 障碍），水路 S04/S05/S09 从无 CLEAR 动作——me 从起点就计划走山路领资源，非水路被挡后被迫绕行。开放问题 1（绕山主动 vs 逼路）**答：主动选择**。

## 3. 净分 ROI 精算

对每局投影"若走最优水路、放弃绕行资源/任务"的终局分（`rules.py` 镜像，假设级上界——不含处理/天气停靠损耗，故对"不绕行"乐观）：

- 放弃 ICE_BOX：端鲜度 −10、好果多转坏 1 篓（80 阈值）。
- 放弃 SHORT_HORSE：移动帧回升、损耗回升。
- 放弃 off-route 任务（S08 的 3 个任务）：task_base 按领任务数等比扣减。

代表性败局 `2696_vs_2986`：

| | 实际（绕行） | 投影（不绕行） | Δ |
|---|---|---|---|
| 端鲜度 | 83.36 | 83.02 | −0.34 |
| 好果 | 97 | 99 | +2 |
| task_base | 150 | 60（5 任务 forfeit 3） | −90 |
| 任务分 | 180 | 75 | −105 |
| 送达分 | 240 | 200 | −40 |
| 鲜度分 | 150 | 149 | −1 |
| **总分** | **766** | **616** | **−150** |

**绕行净增益 +150**。机制：冰鉴 +10 端鲜度恰好抵消山路多损的 4.9（端鲜度基本持平 83.36 vs 83.02），而绕行额外赚了 3 个 off-route 任务（+105 任务分 +40 送达分）+ 马。**鲜度是 wash，任务是纯赚。**

### 群体 ROI（64 交付局）

- 绕行净增益均值 **+93**、中位 **+147**。
- 仅 3 局"绕行 hurts"——均为未交付局（me 卡死、实际分 ~60，投影的"不绕行能交付"是假象，失真不可信）。
- **64/64 交付局绕行净增益 > 0**。

## 4. 决策点定位

绕行由 `client/strategy/decision.py` 的贪心瀑布 `_plan` 发起：起点即 `CLEAR S06` + `MOVE S06`，动机是 `_maybe_claim`（ICE_BOX 机会式领取）+ `_maybe_task`（S08 任务绕路）。**这不是 bug——贪心瀑布在此局做了 +150 的正确选择**：它识别到 S06 冰鉴 + S08 马/任务的山路绕行，其净收益（任务 +105 + 送达 +40）远大于多损的 4.9 鲜度（−8.8 分）。

## 5. 结论与对 Iter 35 计划的修正

### 5.1 三项硬结论

1. **Iter 34 勘误自身勘误**：`on=` 是对手节点，me 实际走 `S01→S06` 直连山路（非 S01→S02→S03→S06）。绕行成本 +7 帧 / +4.895 损耗（非 +64/+7.43）。
2. **绕行是净正收益（+147 中位）**：冰鉴 +10 抵消多损的 4.9（鲜度 wash），另赚 3 off-route 任务 + 马。64/64 交付局绕行净增益 > 0。
3. **路线非 quality-route 鲜度 gap 根因**：me 端鲜度 82.6 vs opp 93.2 的 gap 是**操作层**的——对手在低损耗水路上全程用 FAST_HORSE（base 1200 → 帧 −20% → 损耗 −20%），total_loss 13.79；me 在山路绕行 + 仅 S08-S10 段用 SHORT_HORSE，total_loss 26.64。gap 来自**马的密度 + 路线基础损耗**，不来自"绕行"决策本身（绕行反而在鲜度上 wash、在任务上大赚）。

### 5.3 真正的杠杆（提上 Iter 36）

quality-route 鲜度 gap 的真因是**马的密度**：对手全程 FAST_HORSE、me 仅 1 匹 SHORT_HORSE 用 14 帧。Iter 34 §4 候选 A（马匹密集领取）即此——但须 codeagent 真实 A/B N≥30 正向才合入，不在 Iter 35 范围。

## 6. 开放问题与置信度

- **对手路线低置信**：`on=` 稀疏（≤24 帧、仅 oppNode 变化点）。重建对手走 S01→S02→S03→S07→S09→S10→S11→S14→S15（ROAD 路，360 帧 / 20.04 损耗），但对手实际 total_loss 13.79 < 该路无马损耗 20.04，**仅当对手全程 FAST_HORSE（→ ~16.7）+ 冰鉴才自洽**，与水路 + 马（→ 14.1）+ 冰鉴（→ 96.2）也自洽。两者无法从稀疏数据区分。**结论：对手路线不可确证，但其低损耗可由"低损耗路线 + 全程马"解释，与 me 的绕行无关。**
- **ROI 为上界乐观估计**：投影不含处理/天气停靠损耗，故"不绕行"的真实分只会更低、绕行净增益只会更大。结论方向（绕行净正）稳健。
- me 路线固定点假设：64/67 局完全一致，3 局偏离均为未交付卡死，非路线选择差异。稳健。
