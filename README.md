# Claude Code Agent 初始化说明

## 项目目标

设计并实现一个 **Agent 驱动的软件大赛交付件开发系统**。

该 Agent 不仅能够开发符合比赛要求的交付件，还能够基于比赛运行日志不断分析、迭代、优化交付件，实现持续演进。

---

# Agent 的核心职责

整个 Agent 系统需要包含以下四个能力模块：

```
                ┌────────────────────┐
                │     CLAUDE.md      │
                │ 项目能力基线（知识） │
                └─────────┬──────────┘
                          │
        ┌─────────────────┼──────────────────┐
        │                 │                  │
        ▼                 ▼                  ▼
  交付件规格基线      Client实现框架      Log分析迭代框架
        │                 │                  │
        └─────────────────┼──────────────────┘
                          ▼
                 能力持续演进（迭代）
```

---

# 1. Agent 项目能力基线

Agent 需要维护一份完整的 **CLAUDE.md**，作为整个项目的唯一能力基线（Single Source of Truth）。

CLAUDE.md 应详细记录：

- 项目目标
- 当前系统架构
- Agent 的职责
- 已实现能力
- 未实现能力
- 当前存在的问题
- 已知限制
- 后续规划
- 每轮迭代的能力变化

要求：

- CLAUDE.md 必须始终保持最新。
- 每次能力发生变化时同步更新。
- Agent 的所有开发工作均以 CLAUDE.md 为基础。

---

# 2. 交付件规格基线

建立一套完整的 **Delivery Specification（交付件规格）**。

其作用：

> 明确比赛交付件必须满足的所有要求，作为代码实现的依据。

内容至少包括：

## 功能要求

- 功能列表
- 输入
- 输出
- 生命周期

## 通信协议

严格遵循比赛提供的：

- 任务书： 一骑红尘：荔枝争运战 参赛选手任务书.md

- 通信协议：一骑红尘：荔枝争运战 通信协议.md

  


所有实现必须严格符合上述文档。

---

## 性能要求

例如：

- 响应时间
- 内存限制
- CPU限制
- 网络限制
- 超时策略

---

## 工程要求

例如：

- Python版本
- 第三方依赖
- 文件结构
- 配置方式
- 日志格式
- 启动方式

---

# 3. Client（交付件）实现框架

使用 **Python** 实现最终提交至比赛平台的交付件。

要求：

- 完全符合交付件规格基线
- 严格遵守比赛任务书
- 严格遵守通信协议
- 可直接作为比赛提交物

建议工程结构：

```text
client/
│
├── main.py
├── config.py
├── protocol/
├── core/
├── strategy/
├── communication/
├── utils/
├── logger/
└── tests/
```

其中需要包含：

## 通信模块

负责：

- 建立连接
- 消息发送
- 消息接收
- 心跳
- 异常恢复

---

## Strategy

实现比赛策略。

要求：

策略与通信解耦。

---

## 配置模块

统一管理：

- 参数
- 环境变量
- 超时时间
- 调试开关

---

## Logger

交付件必须具备完整日志能力。

日志需要记录：

- 收到的数据
- 发出的数据
- 决策过程
- 关键状态
- 错误信息
- 异常信息
- 时间消耗

日志必须方便赛后分析。

---

# 4. Log 分析与迭代框架

由于比赛平台运行在公司内网，Agent 无法直接访问运行环境。

因此：

每次比赛结束后，将运行日志导出，并放入项目指定目录。

例如：

```text
logs/
    match_001/
    match_002/
    match_003/
```

要求：

每场比赛日志必须隔离保存。

---

## Agent 日志分析能力

Agent 需要能够自动完成：

### 日志解析

分析：

- 通信过程
- 状态变化
- 决策路径
- 异常情况
- 超时情况
- 崩溃原因

---

### 效果评估

评估：

- 当前策略表现
- 成功原因
- 失败原因
- 性能瓶颈

输出：

- 优点
- 问题
- 风险

---

### 改进建议

根据日志：

自动提出：

- 策略优化建议
- 架构优化建议
- Bug 修复建议
- 参数调整建议

---

### 自动迭代

根据分析结果：

更新：

- Client实现
- CLAUDE.md
- Delivery Specification

保证整个项目不断演进。

---

## 能力沉淀

每完成一次日志分析，都需要形成能力沉淀。

例如：

```text
logs/
    match_005/
        analysis.md
```

同时更新：

- CLAUDE.md
- CHANGELOG.md
- 经验总结

要求：

所有经验必须形成长期知识，而不是仅停留在日志中。

# 项目建议目录结构

```text
project/
│
├── CLAUDE.md                  # Agent 能力基线（SSOT）
├── README.md
├── CHANGELOG.md
│
├── docs/
│   ├── delivery_spec.md
│   ├── protocol.md
│   ├── task.md
│   └── architecture.md
│
├── client/                    # ← 交付件根目录：其“内容”手动打成 ZIP 提交
│   ├── start.sh               #   启动脚本（ZIP 根，可执行，无中文）
│   ├── main.py
│   ├── config.py
│   ├── communication/
│   ├── protocol/
│   ├── core/
│   ├── strategy/
│   ├── logger/
│   ├── utils/
│   ├── tests/
│   └── logs/                  #   运行期 trace 日志 match_*.log（gitignore，随包下载回本地）
│
├── logs/                      # 采集目录（client 之外）：取回的 trace 日志入库，Claude 直接分析
│   └── README.md
│
├── samples/                   # 参考样例（只读，运行期不 import）
└── scripts/
    └── mock_server.py         # 本地假服务端（离线回归）
```

> 打包：仓库不含打包脚本，由人工把 `client/` 的**内容**压成 ZIP（`start.sh` 直接位于 ZIP 根），打包前剔除 `__pycache__/`、`logs/*.log`。

# Agent 工作原则

Agent 在整个项目生命周期中必须遵循以下原则：

1. **CLAUDE.md 是唯一能力基线（Single Source of Truth）**。
2. 所有代码实现必须严格遵循交付件规格基线。
3. 所有实现必须严格符合比赛任务书和通信协议。
4. 每场比赛日志必须独立保存，确保可追溯。
5. 每轮日志分析必须输出完整分析报告。
6. 每轮优化后必须同步更新能力基线（CLAUDE.md）。
7. 所有经验、问题与改进建议都应沉淀为长期知识，形成持续迭代的闭环开发体系。
